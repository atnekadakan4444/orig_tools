# 汎用ディープ調査（fact × pain の並列調査）

指定された **1 テーマ**について、独立した 2 つの調査サブエージェントを `delegate_task` で並列に起動し、
それぞれが実ページの本文に基づいて調査して **別々のファイル**に保存する。

- **fact サブ**: そのテーマの「現状・論点・事実（何が起きているか / 何が争点か / 数字や事例）」を調べる。
- **pain サブ**: そのテーマの当事者の「困りごと・面倒・非効率」を調べる。

## 役割分担（重要）

このスキルの **メイン（オーケストレータ）は、2 体のサブを起動して保存先を報告するだけ**。
**集約・分析・企画案・MVP 案は行わない**。各サブは自分の担当を正確に調べ、自分でファイルに保存する。
最終的なまとめや解釈が必要なら、保存された 2 ファイルを人／別担当が読んで行う。

## 対象テーマ

呼び出し時に指定される。ジャンルは問わない（例: 「LLM のローカル実行環境の現状」「電動キックボードの規制」
「介護記録の管理」「町内会の運営」など）。指定が無ければユーザーに確認する。
**1 回の実行は 1 テーマに限定する**（複数テーマ同時はモデルが破綻し、URL 捏造や検索スニペットへの退行を招く）。

## 手順（メインが行うこと）

1. 本日の日付とテーマスラッグを決める（英小文字・ハイフン。例: `local-llm` / `kickboard-rule`）。
2. `delegate_task` で **2 体を並列起動**する（`background: true` で 2 回投入、または複数 goal のバッチ投入）。
   - 並列上限は `delegation.max_concurrent_children`（デフォルト 3）。2 体なので問題ない。
   - `toolsets` は **省略**して親から継承させる（Web 検索・browser・write_file が使えること）。
   - 各サブの `goal` には、担当（fact / pain）・対象テーマ・保存先パス・下記の「各サブへの厳守事項」を明記する。
3. 2 体の完了を待ち、返ってきた `status` と `summary`、および **保存先 2 パス**を報告する。
   メイン自身は本文を精読し直したり、内容を要約・分析したりしない。

### fact サブに渡す goal の骨子

```
あなたは「<テーマ>」の現状・論点・事実を調べる調査担当です。
web_search で「<テーマ> 現状」「<テーマ> 最新」「<テーマ> 課題」「<テーマ> 仕組み」等を複数試し、
有望な実ページ（公式情報 / ニュース / 解説記事 / note / 個人ブログ / まとめ）を browser で実際に開いて
本文を最低 3 ページ、可能なら 5 ページまで精読する。本文に書かれている事実・論点・数字・事例だけを、
原文のニュアンスを保って抽出する。下記の厳守事項に従い、write_file で指定パスに保存せよ。
保存先: /opt/data/obsidian/deep-research/<スラッグ>/<本日日付>_<テーマ>_fact.md
```

### pain サブに渡す goal の骨子

```
あなたは「<テーマ>」の当事者の困りごと・面倒・非効率を調べる調査担当です。
web_search で「<テーマ> 困っている」「<テーマ> 面倒」「<テーマ> 不満」「<テーマ> 限界」等を複数試し、
当事者の悩みが書かれていそうな実ページ（note / はてなブログ / はてな匿名ダイアリー / Yahoo!知恵袋 /
個人ブログ / まとめ）を browser で実際に開いて本文を最低 3 ページ、可能なら 5 ページまで精読する。
本文に実際に書かれている困りごとを、原文のニュアンスを保って具体的に抽出する。下記の厳守事項に従い、
write_file で指定パスに保存せよ。
保存先: /opt/data/obsidian/deep-research/<スラッグ>/<本日日付>_<テーマ>_pain.md
```

## 各サブへの厳守事項（両サブ共通・goal に必ず含める）

- 本文取得は **`browser` で実際にページを開いて読む**。**`web_extract` は本環境の SearXNG 構成（検索専用）では使えず必ず失敗するので呼ばない**（無駄な失敗がループ警告を招く）。`web_search` は検索（URL 探し）にのみ使う。
- browser で実際に開いて本文を読めたページの内容だけを使う。読めなかったページの内容を推測で書かない。
- URL は実際に開いたものをそのまま記載する。**プレースホルダ（`xxxx` 等）や「検索結果より」という記載は禁止**。
- 事実・引用・URL を創作しない。件数は実際に読めたページ由来の数に従い、水増ししない。
- X（旧 Twitter）と Reddit はログイン必須で本文が読めないため、読めなければ触れない。
- 企画案・MVP 案・解決策の提案は書かない（調査に徹する）。
- 保存は必ず `write_file`。サブエージェント内では `memory` 書き込みや `delegate_task`（再帰委譲）はブロックされる。

## 出力

2 ファイルを別々に保存する（親フォルダが無ければ自動生成される）。

- fact: `/opt/data/obsidian/deep-research/<スラッグ>/<本日日付>_<テーマ>_fact.md`
- pain: `/opt/data/obsidian/deep-research/<スラッグ>/<本日日付>_<テーマ>_pain.md`

各ファイルの frontmatter:

```
tags: [hermes, 調査, <テーマ>, <fact または pain>]
date: <本日日付>
source: browser
```

本文は「具体的内容（本文に基づく）／出典 URL（実際に開いたもの）／ページタイトル」を 1 セットで列挙し、
末尾に「実際に本文を読めたページ数」と「読めなかった/除外したページ」を正直に記載する。

## 注意（実測で判明した落とし穴・前提）

- **非対話実行（`hermes -z` / cron）でサブのツールを動かすには `delegation.subagent_auto_approve: true` が必要**。
  承認コールバックはスレッドローカルを継承せず、デフォルト（`false`）だとサブの browser/write_file が自動拒否される。
  対話実行では都度承認できるため未設定でも動くが、無人運用するなら `config.yaml` の `delegation:` に追記する。
- cron のトップレベルから `delegate_task` が使えるかは環境依存で要検証。まずは対話実行で挙動を確認してから
  無人化すること。単一エージェントで確実に回したい場合は、fact / pain を 1 体が順に調べる逐次型に落とす。
- **1 回の実行は 1 テーマに限定**。複数テーマ同時は破綻の元。
- web_search が辞書・用語解説ページばかり返す場合は SearXNG のエンジン劣化を疑う
  （`hermes_agent/searxng/settings.yml` を確認。CAPTCHA 常習エンジンは無効化済み、Bing 主軸）。
- 1 テーマあたり各サブ 20〜30 分程度かかる。並列なので全体はおおむねその範囲に収まる。
