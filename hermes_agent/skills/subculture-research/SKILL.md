# サブカル領域 ペイン調査

指定された **1 領域**について、当事者の「困りごと・面倒・非効率」を実ページの本文に基づいて調査し、
調査素材として Obsidian に保存する。

## 役割分担（重要）

このスキルは **調査だけ**を行う。最終的な分析・まとめ・企画案の作成は別の担当（Claude）が行うため、
**企画案・MVP 案・解決策は書かない**。素材を正確に集めることに徹する。

## 対象領域

呼び出し時に指定される。例: 推し活 / 同人活動 / トレーディングカードゲーム / VTuber視聴・切り抜き制作 /
コスプレ / ボカロ など。指定が無ければユーザーに確認する。

## 手順

1. `web_search` で「<領域> 管理 面倒」「<領域> アプリ 欲しい」「<領域> エクセル 限界」
   「<領域> 困っている」等のクエリを複数試し、当事者の悩みが書かれていそうな実ページ
   （note / はてなブログ / はてな匿名ダイアリー / Yahoo!知恵袋 / 個人ブログ / まとめ記事）を探す。
2. 有望なページを `browser` ツールで実際に開き、本文を読む。**最低 3 ページ、可能なら 5 ページまで精読**する。
3. 本文に実際に書かれている「困りごと・面倒・非効率」を、原文のニュアンスを保って具体的に抽出する。

## 厳守事項

- 本文取得は **`browser` で実際にページを開いて読む**。**`web_extract` は本環境の SearXNG 構成（検索専用）では使えず必ず失敗するので呼ばない**（無駄な失敗がループ警告を招く）。`web_search` は検索（URL 探し）にのみ使う。
- browser で実際に開いて本文を読めたページの内容だけを使う。読めなかったページの内容を推測で書かない。
- URL は実際に開いたものをそのまま記載する。**プレースホルダ（`xxxx` 等）や「検索結果より」という記載は禁止**。
- 事実・引用・URL を創作しない。
- X（旧 Twitter）と Reddit はログイン必須で本文が読めないため、読めなければ触れない。
- 件数は実際に読めたページ由来の数に従う。水増ししない。
- 企画案・MVP 案・解決策の提案は書かない。

## 出力

`/opt/data/obsidian/deep-research/<領域スラッグ>/<本日日付>_<領域>調査.md` に `write_file` で保存する
（親フォルダが無ければ自動生成される）。領域スラッグの例: `oshikatsu` / `doujin` / `tcg` / `vtuber`。

frontmatter:

```
tags: [hermes, 調査, <領域>]
date: <本日日付>
source: browser
```

各項目は「困りごとの具体的内容（本文に基づく）／出典 URL（実際に開いたもの）／ページタイトル」を
セットで列挙する。末尾に「実際に本文を読めたページ数」と「読めなかった/除外したページ」を正直に記載する。

## 注意（実測で判明した落とし穴）

- **1 回の実行は 1 領域に限定する**。複数領域を同時に扱うとモデルが破綻し、検索スニペットへの退行や
  URL 捏造（`xxxx` 等）が起きる。複数領域を調べたい場合は領域ごとに分けて呼び出す。
- 無人実行（cron）では `execute_code` がブロックされるため、保存は必ず `write_file` を使う。
- 1 領域あたり 20〜30 分程度かかる。
