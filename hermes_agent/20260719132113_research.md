# Hermes Agent × ローカルOllama 連携調査レポート

- 調査日: 2026-07-19
- 対象: `hermes_agent/docker-compose.yml`（コンテナ名 `hermes-sandbox`, image `nousresearch/hermes-agent:latest`）
- 症状: ローカルPCで稼働中のOllamaとコンテナが連携できない

## 結論（サマリ）

ネットワークは最初から正常だった。原因は **設定の読ませ方**。Hermes Agent は
使用モデル・エンドポイントを `/opt/data/config.yaml` からのみ読み込む設計であり、
`docker-compose.yml` に書かれていた `OPENAI_BASE_URL` / `MODEL` 等はHermesが参照しない
変数名のため **すべて無視** されていた。その結果 config.yaml のデフォルト
（`anthropic/claude-opus-4.6` + OpenRouter）のまま動作していた。

config.yaml をOllama向けに修正し、`/opt/data` を永続化することで解決。
Hermes経由の推論をエンドツーエンドで確認済み。

## 切り分けの経緯

### 1. Ollama側の状態（ホスト）

- `ollama serve` が `*:11434`（全インターフェース）でLISTEN。
- `qwen3.6:latest`（36B, Q4_K_M）が存在。
- `curl http://localhost:11434/api/tags` 正常応答。

### 2. コンテナ → Ollama の疎通

- コンテナは稼働中（`Up`）。
- コンテナ内から `http://host.docker.internal:11434/api/tags` → 正常応答。
- `/v1/models` → `qwen3.6:latest` を返す。
- `/v1/chat/completions`（`Authorization: Bearer ollama`）→ 正常に補完を返す。

**→ ネットワーク・OpenAI互換API・モデル存在・認証、すべて問題なし。**

### 3. 実際に使われているモデルの確認

コンテナのTUIバナーおよびステータス行が **`claude-opus-4.6 · Nous Research`** を表示。
Ollamaのqwenモデルを使っていなかった。

### 4. Hermesの設定読み込みロジック調査

`/opt/hermes/cli.py` を確認した結果、以下が判明。

- **モデル/base_url/provider は `config.yaml` が唯一の権威**。
  実装コメントに明記:
  > `LLM_MODEL/OPENAI_MODEL env vars are NOT checked — config.yaml is authoritative.`
  （マルチエージェント構成でenvが互いに競合するのを避けるため）
- Hermesが実際に読むenv（参考）:
  - base_url → `OPENROUTER_BASE_URL`
  - provider → `HERMES_INFERENCE_PROVIDER`
  - api_key → `OPENAI_API_KEY`
  - 最大反復 → `HERMES_MAX_ITERATIONS`（`MAX_ITERATIONS` ではない）
  - ただし **モデル名はenv不可**。
- config探索順: `${HERMES_HOME}/config.yaml`（優先） → プロジェクトの `cli-config.yaml`。
  本コンテナは `HERMES_HOME=/opt/data`。

### 5. 元 docker-compose.yml の無効な設定

以下はHermesが参照しない変数名のため、書いても無視されていた:

- `OPENAI_BASE_URL` / `OPENAI_API_BASE`
- `MODEL` / `OPENAI_MODEL_NAME` / `LITELLM_MODEL`
- `LLM_PROVIDER`
- `MAX_ITERATIONS`

### 6. config.yaml の実際の中身（修正前）

```yaml
model:
  default: anthropic/claude-opus-4.6
  provider: auto
  base_url: https://openrouter.ai/api/v1
```

これがコンテナ初回起動時に自動生成されたデフォルト。`/opt/data` はボリューム
未マウントだったため、コンテナ再生成のたびにこの初期値へ戻る状態だった。

## 根本原因

Hermes Agent はモデル・エンドポイントを `config.yaml` からのみ読み込むが、
compose では代わりに（Hermesが無視する）`OPENAI_*` 系envを設定していた。
このため config.yaml のデフォルト（OpenRouter経由の `claude-opus-4.6`）のまま動作し、
ローカルOllamaが一切使われていなかった。ネットワーク不通が原因ではない。

## 対応内容

### 1. config.yaml をOllama向けに修正

`provider: ollama` は内部的に custom 扱いとなり、明示した `base_url` が尊重される
（Dockerの `host.docker.internal` でも localhost に上書きされない。GitHub #27132 対応済み）。

```yaml
model:
  default: qwen3.6:latest
  provider: ollama
  base_url: http://host.docker.internal:11434/v1
```

### 2. コンテナ再起動 → 反映確認

TUIバナーが `claude-opus-4.6` から **`qwen3.6:latest`** に変化。

### 3. 設定の永続化（名前付きボリュームへ移行）

`/opt/data`（56MB）を外部名前付きボリューム `hermes-agent-data` へ移行。
これにより `docker compose down/up`（コンテナ再生成）後も、モデル設定・memories・
sessions が維持され、手動での再設定が不要になる。

### 4. docker-compose.yml の修正

- 無効な `OPENAI_*` / `MODEL` / `LLM_PROVIDER` / `MAX_ITERATIONS` envを削除
- `MAX_ITERATIONS` → `HERMES_MAX_ITERATIONS`（正しい変数名）へ修正
- workspaceパスのtypo `hermos_agent` → `hermes_agent` を修正
- `extra_hosts: host.docker.internal:host-gateway` を追加（環境差異・Linux対策）
- `hermes-agent-data:/opt/data`（external volume）を追加して永続化

### 5. エンドツーエンド検証

```
docker exec hermes-sandbox hermes -z "Reply with exactly: OLLAMA_OK" --yolo
→ OLLAMA_OK
```

Hermesが実際にローカルOllamaで推論し、正しく応答することを確認。

## 補足・注意点

- モデルを変更したい場合は今後 `config.yaml` を編集するか、TUIの `/model` コマンドで行う
  （envでは変更できない）。
- `network_mode: "none"` は Ollama連携に `host.docker.internal` 到達が必要なため使えない
  （compose内にコメントで明記）。
- 現在のコンテナはそのまま稼働継続で問題ない。次回 `docker compose up -d` での再生成時に
  外部ボリューム側の設定が使われ、手動設定なしで連携が維持される。
