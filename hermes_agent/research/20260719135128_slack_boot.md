# hermes-agent: Slackから疎通できない問題の調査と対策

- 日付: 2026-07-19
- 対象: `hermes_agent`（Docker / Ollama / Slack連携）
- 症状: hermes-agent コンテナは正常に立ち上がるが、**Slackから疎通できない**。しかもログにエラーが出ないため原因が掴めない。

## 結論（TL;DR）

**Slackイベントを待ち受ける gateway（メッセージングゲートウェイ）プロセスが起動していなかった**ことが原因。

- コンテナ起動時は既定で**対話CLI（TUI）だけ**が立ち上がる。
- Slackを待ち受けるのは別プロセス `hermes gateway run`。これが停止していた。
- 待ち受け側が存在しないため、Slackからメッセージを送っても**そもそも何も受信処理が走らず、エラーログすら出ない**。

トークン類（`SLACK_BOT_TOKEN` / `SLACK_APP_TOKEN` / `SLACK_ALLOWED_USERS`）やOllama設定は正しく、gatewayを起動しただけで即 `slack: connected` になった。

## 調査の経緯

### 1. コンテナの起動状態を確認

`docker logs hermes-sandbox` の出力は Hermes Agent の**対話CLI（TUI）のウェルカム画面**だった。
`docker-compose.yml` は `tty: true` / `stdin_open: true` のみで `command:` を上書きしていないため、
イメージの既定エントリポイント（＝対話CLI）だけが起動している状態。Slack連携の待ち受けは含まれない。

### 2. Slackの待ち受け方法を特定

`hermes --help` および各サブコマンドを確認:

- `hermes slack` … `manifest` 生成のみ（待ち受けではない）
- `hermes gateway run` … **メッセージングプラットフォーム（Slack等）＋cronスケジューラを実際に待ち受けるプロセス**

つまりSlack疎通には gateway の常駐が必須。

### 3. gatewayが停止していることを確認

```
$ hermes gateway status
✗ Gateway is not running

$ hermes gateway list
  ✗ default (current)  — not running
```

s6 のサービススロット `gateway-default` は登録されている（`s6-supervise gateway-default` は存在）が、
サービス定義ディレクトリに **`down` ファイル**（s6の「自動起動しない」印）が置かれていた:

```
$ ls -la /run/service/gateway-default/
-rw-r--r-- 1 hermes hermes  0 ... down        ← これ
-rwxr-xr-x 1 hermes hermes ... run            ← 中身は `hermes gateway run --replace`
```

gatewayログ `/opt/data/logs/gateways/default/current` は 0 バイト（一度も起動していない）。

### 4. 仕様上の根拠（イメージの起動契約）

イメージ内 `/opt/hermes/docker/stage2-hook.sh`（L472-497）に明記されている:

> 新規ボリュームには `gateway_state.json` が無いため、boot時のreconciler
> （`cont-init.d/02-reconcile-profiles` → `container_boot.reconcile_profile_gateways`）は
> `gateway-default` の s6 スロットを登録するが **DOWN のままにする**。
> 直近の記録状態が `"running"` のときだけ自動起動する。
> よって**新規ボリュームではgatewayはDOWNで立ち上がり、誰かが手動起動するまでSlackの待ち受けが始まらない**。
> フレッシュなボリュームで初回bootから起動したいオーケストレーターは
> `HERMES_GATEWAY_BOOTSTRAP_STATE=running` を設定すればよい（state fileを種付けする）。

これが今回の状況と完全に一致。今回は新規の外部ボリューム `hermes-agent-data` を使っており、
過去に一度も gateway を「running」で記録していなかったため、boot時にDOWNのまま放置されていた。

### 5. 手動起動で切り分け（＝原因確定）

```
$ hermes gateway run
→ gateway is now running under s6 supervision ...

$ hermes gateway status
✓ Gateway is running (PID: 292)

$ cat /opt/data/gateway_state.json
{"desired_state":"running", ...,
 "platforms":{"slack":{"state":"connected","error_code":null,"error_message":null,...}}, ...}
```

gatewayを起動しただけで **Slackが `connected`** になった。トークン・Ollama設定は問題なし。
→ 原因は「gateway未起動」で確定。

## なぜ「エラーが出ない」のか

Slackの待ち受けプロセス（gateway）自体が存在しないため、
Slack側からのイベントを**受け取る主体がいない**。
接続失敗や認証エラーのような「動いているものが失敗する」事象ではなく、
「待ち受けが最初から起動していない」ため、コンテナのログにも何も残らなかった。
これが「ログからエラーが確認できず原因不明」に見えた理由。

## 対策

### 恒久対策（実施済み）

`docker-compose.yml` の `environment` に次を追加した:

```yaml
      - HERMES_GATEWAY_BOOTSTRAP_STATE=running
```

- `stage2-hook.sh` が boot時（reconcilerより前）に `gateway_state.json` を `running` で種付けする。
- reconciler が `prior_state=running` を見て gateway スロットを起動 → **初回bootからSlack接続が自動確立**。
- 種付けは `gateway_state.json` が未作成の**初回のみ**（`[ ! -f ]` ガード）。
  以降は永続化された状態が優先されるため、意図的に停止したgatewayが再起動で勝手に上がることはない。

### 即時復旧（実施済み）

コンテナ内で `hermes gateway run` を手動実行し、gatewayを起動済み。
現時点でSlackは `connected`。すぐに利用可能。

## 検証手順（推奨）

現在動いている gateway は手動起動（`Running manually`）のため、
宣言的な自動起動が効くことを確認するために一度クリーンに再起動しておくと確実:

```bash
cd hermes_agent
docker compose down && docker compose up -d
# 数十秒待ってから
docker exec hermes-sandbox hermes gateway status
# → ✓ Gateway is running であること
```

Slackで実際にメッセージを送り、応答が返れば完了。

## 補足・今後のためのメモ

- モデル・エンドポイントは `config.yaml` が唯一の権威（`model.default: qwen3.6:latest` / `provider: ollama` /
  `base_url: http://host.docker.internal:11434/v1`）。env の `OPENAI_API_KEY=ollama` はダミーで問題なし。
- gateway を「サービスとして」常駐させたい場合は `hermes gateway install`（systemd/launchd）もあるが、
  本構成はDockerなので **`HERMES_GATEWAY_BOOTSTRAP_STATE=running` + s6 supervision** が素直で推奨。
- gateway が上がっているのに疎通しない場合の次の切り分け:
  - `cat /opt/data/gateway_state.json` の `platforms.slack.state` と `error_message`
  - `/opt/data/logs/gateways/default/current` のログ
  - Slackアプリ側（Socket Mode有効、App-Level Token のスコープ `connections:write`、Bot Token スコープ、イベント購読）
