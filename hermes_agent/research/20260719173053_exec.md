# hermes-agent: install.sh 実行によるNous汚染とその復旧

- 日付: 2026-07-19
- 対象: `hermes_agent`（Docker / 公式イメージ `nousresearch/hermes-agent:latest` / Ollama運用）
- 症状: `curl -fsSL https://hermes-agent.nousresearch.com/install.sh | bash` を**コンテナ内で（`docker exec`で）**実行したところ、意図せずNousログインが走り、以降ログに `auth.json` の Permission denied や「Nous Portalにログインしていない」エラーが多発するようになった。

## 結論（TL;DR）

**「CLIが入っていない」わけではなかった。** hermes CLI は公式イメージに最初から同梱されており（`/opt/hermes/bin/hermes`・`/usr/local/bin/hermes`、v0.18.2）、PATHも通っている。install.sh の再インストールは不要だった。

問題は install.sh の**セットアップ・ウィザードが2つの汚染を起こした**こと:

1. **bind mountされたホストの `config.yaml` を丸ごと上書き**し、`provider: ollama`（qwen3.6, host.docker.internal:11434）を `provider: nous`（tencent/hy3:free, nousresearch.com）に変えた。これによりNous認証が要求されるようになった（＝「Nousログインになってしまった」正体）。
2. `docker exec` は**デフォルトでrootとして入る**ため、その中で走ったNousログインが `hermes` ユーザーのHOME（`/opt/data`）内に **root所有のファイル**を書き込んだ。hermesサービスは uid 10000（`hermes`ユーザー）で動くため、root所有の 0600 ファイルを読めず `[Errno 13] Permission denied` を連発した。

## 原因の詳細

### 前提: hermesユーザーのHOMEは永続ボリューム `/opt/data`

```
$ getent passwd hermes
hermes:x:10000:10000::/opt/data:/bin/sh
```

- `/opt/data` は compose の名前付きボリューム `hermes-agent-data`（external）でマウントされ**永続化**される。
- `config.yaml` は `./config.yaml:/opt/data/config.yaml` として**ホストからbind mount**されている（モデル設定はconfig.yamlが唯一の権威という仕様のため）。
- つまり install.sh の書き込みは**コンテナ再作成では消えない**（ボリューム内 & bind mount元のホストファイル）。

### 汚染1: config.yaml の上書き

git コミット済み（本来）と、ウィザード上書き後の比較:

| 項目 | 本来（git HEAD） | ウィザード上書き後 |
|---|---|---|
| provider | `ollama` | `nous` |
| base_url | `http://host.docker.internal:11434/v1` | `https://inference-api.nousresearch.com/v1` |
| default model | `qwen3.6:latest` | `tencent/hy3:free` |
| web(SearXNG) / guardrails / compression / memory / session_reset 等 | 全部あり | **全部消滅**（最小構成に置換） |

### 汚染2: root所有ファイルの生成

Nousログイン実行時刻（08:05前後）にrootで作られたもの:

| パス | 内容 |
|---|---|
| `/opt/data/auth.json`（root:root 0600） | 認証ストア。サービスが読めず Permission denied の主因 |
| `/opt/data/shared/nous_auth.json`（root:root） | Nousの認証トークン本体 |
| `/opt/data/cache/nous_recommended_cache.json`, `model_catalog.json`（root） | Nous向けキャッシュ |
| `/opt/data/config.yaml.bak.*`（root, 08:05〜08:11） | ウィザードがconfigを書き換えた形跡 |

※ `bin/`（uv, uvx, tirith）は Jul 15/04:04 のイメージ由来で、install.sh の痕跡ではない。

### 連鎖したログエラー

```
WARNING hermes_cli.auth: auth: failed to parse /opt/data/auth.json
  ([Errno 13] Permission denied: '/opt/data/auth.json') — starting with empty store.
WARNING gateway.run: Primary provider auth failed: Hermes is not logged into Nous Portal. — trying fallback
WARNING agent.auxiliary_client: Auxiliary Nous client unavailable: no Nous authentication found
```

## 復旧手順（実施済み）

事前に現状（Nous版config と各認証ファイル）を `scratchpad/hermes-revert-backup/` へ退避してから実施。

1. **config.yaml を git から復元**（Ollama構成・SearXNG・guardrails等が全部戻る）
   ```
   git checkout HEAD -- hermes_agent/config.yaml
   ```
2. **root所有のNous認証ファイルを削除**（Ollama運用では不要）
   ```
   docker exec hermes-sandbox bash -lc 'rm -f \
     /opt/data/auth.json \
     /opt/data/shared/nous_auth.json /opt/data/shared/nous_auth.lock \
     /opt/data/cache/nous_recommended_cache.json /opt/data/cache/model_catalog.json'
   ```
3. **コンテナ再起動**（config.yaml は boot 時読み込みのため）
   ```
   docker compose restart hermes-agent
   ```

### 復旧結果

再起動後ログで、今回の汚染起因エラーはすべて0件:

| 項目 | 復旧前 | 復旧後 |
|---|---|---|
| `Permission denied: auth.json` | 多発 | 0件 |
| `not logged into Nous Portal` / `auth: failed to parse` | 多発 | 0件 |
| Nous client unavailable / payment系 warning | 多発 | 0件 |
| config provider | `nous` | `ollama`（qwen3.6） |

- CLI（v0.18.2）はイメージ同梱のまま。再インストール不要。
- `auth.json` は現在存在しないが、Ollama運用では認証不要なので問題なし（必要時にサービスが `hermes` 所有で自動生成する）。

## 教訓・再発防止

- **`docker exec` はデフォルトでrootで入る。** コンテナ内で `hermes auth` やインストール系を走らせると、`hermes` ユーザーのHOME（`/opt/data`）にroot所有ファイルができて Permission denied を再発させる。コンテナ内で hermes を触るときは必ず:
  ```
  docker exec -u hermes hermes-sandbox <command>
  ```
- **CLIは公式イメージに同梱済み。** `command not found` が出ても install.sh で入れ直さない。まず `docker exec -u hermes hermes-sandbox bash -lc 'command -v hermes'` で存在確認する（一時的なPATHリセットで見えないだけのことがある）。
- **install.sh のセットアップ・ウィザードは config.yaml を上書きする。** モデル設定を宣言的に管理している本構成では、実行するなら `| bash -s -- --skip-setup` でウィザードを抑止する。そもそもこの環境ではローカルOllama運用のため install.sh 自体が不要。
- config.yaml は git で版管理しているので、壊れても `git checkout HEAD -- hermes_agent/config.yaml` で即復旧できる。
