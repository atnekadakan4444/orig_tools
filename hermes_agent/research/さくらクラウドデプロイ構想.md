# さくらクラウド デプロイ構想（A案：GPU VM に丸ごと移植）

Mac 上で動いている `hermes_agent`（docker-compose）を、さくらのクラウドの GPU VM
（**高火力 VRT**）1 台にほぼそのまま移植する手順書。

- 方針: 現状の `docker-compose.yml` を極力そのまま使い、変更点を最小化する
- 予算: **2 万円**（短期運用・使い捨て前提）
- 運用期間: 短くてよい（テスト／デモ用途。長期常駐は想定しない）

---

## 1. 結論と全体像

**実現可能。** 唯一の本質的な論点だった「Ollama による LLM 推論」を、さくらの GPU VM
（高火力 VRT）で賄えばよい。コンテナ 2 つ（hermes-agent / searxng）と Slack 連携・
永続ボリュームは Linux VM に Docker を入れればそのまま動く。

### クラウド版アーキテクチャ

```
さくら 高火力 VRT（VM 1 台 / Ubuntu + NVIDIA ドライバ）
┌──────────────────────────────────────────────┐
│  Docker (docker-compose)                       │
│   ┌─ hermes-agent ─┬─ Slack (Socket Mode, out) │──→ 外部: Slack API
│   │                ├─ searxng:8080             │
│   │                └─ host.docker.internal:11434│
│   └─ searxng                                    │
│                         ↓                        │
│  Ollama（VM ホストで直接起動 / GPU 使用）        │
│   OLLAMA_HOST=0.0.0.0:11434  model: qwen3.6      │
└──────────────────────────────────────────────┘
        ↑ inbound は SSH(22) のみ許可
```

**Mac 版との違いは 3 点だけ:**

1. Ollama を「Mac ホスト直起動」→「VM ホスト直起動（`OLLAMA_HOST=0.0.0.0` 必須）」
2. bind mount のパスが Mac 固有 → VM 上のパスに書き換え
3. GPU が Apple Silicon → NVIDIA V100（ドライバは VRT イメージに同梱）

## 2. プラン選定と予算試算（重要）

### 高火力 VRT の料金（2026 年時点・上限付きオンデマンド）

| GPU | VRAM | CPU / メモリ | 時間額 | 日額 | 月額 |
|-----|------|-------------|--------|------|------|
| **V100** | 32GB | 4 コア / 56GB | **481 円** | 11,550 円 | 231,000 円 |
| H100 | 80GB | 24 コア / 240GB | 990 円 | 23,100 円 | 385,000 円 |

- 課金は **起動時間ぶんのみ**。VM を**停止すれば時間課金は止まる**（ディスク等の最小費用は別途少額）
- 月内の起動時間に応じて 時間額 → 日額 → 月額 に自動でキャップされる

### 2 万円での使い方（V100 前提）

- **V100 で約 40 時間ぶん**起動できる（481 円 × 40 ≒ 19,240 円）
- H100 は 990 円/時間で約 20 時間ぶん。qwen3.6 が 32GB に収まるなら **V100 で十分**
- **鉄則: テスト／デモのたびに起動 → 終わったら即停止。** 消し忘れ 1 日 = 最大 11,550 円

> ⚠️ **要事前確認**: `qwen3.6:latest` のパラメータ数・量子化。
> V100 は VRAM 32GB。30B クラス（4bit 量子化で ~20GB 前後）までなら V100 で快適。
> 70B クラスは 32GB に収まらず H100 80GB か、より強い量子化が必要。
> → **Step 0 で必ず実測**して、収まらなければ H100 に切替 or モデルを小さくする。

### （参考）代替案: 高火力 DOK
「Slack が来たときだけ推論」なら、コンテナ従量型の **高火力 DOK**（V100 約 57.6 円/時間〜、
新規 3,000 円無料枠）が安いが、Slack 常駐待受＋compose 構成とは相性を要検討。
本手順は A案（VM 丸ごと移植）に集中する。

## 4. デプロイ手順

### Step 0. モデルの VRAM 要件を先に確認（超重要）

先に手元の Mac で `qwen3.6:latest` の実 VRAM を確認し、V100(32GB) / H100(80GB)
どちらを借りるか決める。

```bash
ollama show qwen3.6:latest          # パラメータ数・量子化・サイズを確認
# 実行中の占有 VRAM を見る（推論を 1 回叩いてから）
ollama ps
```

- モデルサイズ（GB）＋コンテキスト分のバッファで 32GB 未満に収まりそうなら **V100 を選択**
- 収まらない／際どい場合は **H100**、または量子化を強めた小さいタグに変更

### Step 1. 高火力 VRT で VM を作成

コントロールパネルから:

- プラン: **高火力 VRT / V100**（Step 0 の結果次第で H100）
- OS: Ubuntu 22.04 LTS（NVIDIA ドライバ／CUDA 同梱イメージがあればそれを選ぶ）
- ディスク: 100GB 程度（モデル本体で数十 GB 使う想定。qwen3.6 のサイズ次第で増やす）
- SSH 公開鍵: 事前準備の鍵を登録
- ネットワーク: グローバル IP あり（Slack は Socket Mode で outbound のみだが、SSH 用に必要）

作成後、SSH 接続:

```bash
ssh ubuntu@<VMのグローバルIP>
```

### Step 2. 初期セットアップ & ファイアウォール

```bash
sudo apt update && sudo apt -y upgrade

# inbound は SSH のみ許可（Ollama:11434 を絶対に外部公開しない）
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow 22/tcp
sudo ufw enable
sudo ufw status
```

> さくらのクラウド側のパケットフィルタでも「SSH(22) 以外の inbound を拒否」を二重で設定推奨。
> hermes ↔ searxng ↔ Ollama の通信はすべて VM 内部（Docker bridge / loopback）で完結し、
> 外向きに開ける必要はない。

### Step 3. NVIDIA ドライバ確認

```bash
nvidia-smi        # GPU が見えれば OK（V100/H100 が表示される）
```

表示されない場合はドライバ導入（VRT の同梱イメージなら通常は不要）:

```bash
sudo apt -y install nvidia-driver-535   # 例。イメージに応じて調整
sudo reboot
```

### Step 4. Ollama をホストに導入し、モデルを pull

```bash
curl -fsSL https://ollama.com/install.sh | sh

# 【Linux 必須】コンテナから host.docker.internal 経由で届くよう全 IF で LISTEN させる
sudo mkdir -p /etc/systemd/system/ollama.service.d
sudo tee /etc/systemd/system/ollama.service.d/override.conf >/dev/null <<'EOF'
[Service]
Environment="OLLAMA_HOST=0.0.0.0:11434"
EOF
sudo systemctl daemon-reload
sudo systemctl restart ollama

# GPU 稼働の確認とモデル取得
nvidia-smi
ollama pull qwen3.6:latest
ollama run qwen3.6:latest "ping"   # 応答すれば OK。Ctrl+D で抜ける
```

> ⚠️ Mac では `host.docker.internal` はデフォルトで解決できたが、**Linux では Ollama が
> `127.0.0.1` だけで LISTEN していると Docker からの接続が届かない**。上記 `OLLAMA_HOST=0.0.0.0`
> が必須。ufw で 11434 の inbound を塞いでいるので外部露出はしない（Docker bridge は内部扱い）。

### Step 5. Docker / docker-compose 導入

```bash
# Docker Engine + compose plugin
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER
newgrp docker   # 再ログインでも可

docker --version && docker compose version
```

> 本構成では **Ollama をホスト直起動**するため、Docker への GPU passthrough
> （nvidia-container-toolkit）は**不要**。Ollama もコンテナ化したい場合のみ導入する（後述の代替案）。

### Step 6. リポジトリ配置と compose 書き換え

```bash
# 例: /opt/hermes_agent に配置（Git clone でも scp でも可）
sudo mkdir -p /opt/hermes_agent && sudo chown $USER:$USER /opt/hermes_agent
# Mac から: scp -r hermes_agent ubuntu@<IP>:/opt/
cd /opt/hermes_agent
```

`docker-compose.yml` の **bind mount のホストパス**を Mac 固有パスから VM のパスに変更:

```yaml
# 変更前（Mac 固有）
- /Users/nakada/development/orig_tools/hermes_agent:/workspace
# 変更後（VM）
- /opt/hermes_agent:/workspace
```

`config.yaml` の Ollama 向き先は**変更不要**（`http://host.docker.internal:11434/v1` のまま）。
`docker-compose.yml` に `extra_hosts: host.docker.internal:host-gateway` が既にあるため
Linux でも解決される。

### Step 7. `.env` を安全に配置

`.env` は git 管理外なので手動で置く（内容は事前準備で控えたもの）:

```bash
# Mac から scp（推奨。履歴に残さない）
scp hermes_agent/.env ubuntu@<IP>:/opt/hermes_agent/.env
chmod 600 /opt/hermes_agent/.env
```

含めるキー: `SLACK_APP_TOKEN` / `SLACK_BOT_TOKEN` / `SLACK_ALLOWED_USERS` / `SEARXNG_SECRET`

### Step 8. 外部ボリュームを作成（＋任意でデータ移行）

`hermes-agent-data` は compose 上 `external: true` なので、先に作成が必要:

```bash
docker volume create hermes-agent-data
```

Mac の memories / sessions を引き継ぎたい場合（短期用途なら**スキップ可**）:

```bash
# Mac 側でエクスポート
docker run --rm -v hermes-agent-data:/data -v "$PWD":/backup alpine \
  tar czf /backup/hermes-data.tgz -C /data .
scp hermes-data.tgz ubuntu@<IP>:/tmp/
# VM 側でインポート
docker run --rm -v hermes-agent-data:/data -v /tmp:/backup alpine \
  sh -c "cd /data && tar xzf /backup/hermes-data.tgz"
```

### Step 9. SearXNG 設定の確認

`hermes_agent/searxng/settings.yml`（`search.formats` に `json` 追加済み）がリポジトリに
含まれていれば追加作業なし。`SEARXNG_SECRET` は Step 7 の `.env` から注入される。

### Step 10. 起動と疎通確認

```bash
cd /opt/hermes_agent
docker compose up -d
docker compose ps
docker compose logs -f hermes-agent   # gateway が running で Slack 接続するか確認
```

チェックリスト:

- [ ] `nvidia-smi` で `ollama` が GPU を掴んでいる（推論時に VRAM 使用が増える）
- [ ] LLM 疎通: `docker exec hermes-sandbox hermes -z "1+1は？" --yolo` が応答
- [ ] Slack: 許可ユーザーから DM/メンションして応答が返る（Socket Mode）
- [ ] web 検索: Slack 経由で検索を促し、SearXNG 経由の実結果が返る
- [ ] SearXNG デバッグ: VM 内から `curl 'http://127.0.0.1:8888/search?q=test&format=json'`

### Step 11. コスト管理（運用中）

**使わない間は VM を停止して時間課金を止める。** これが 2 万円運用の生命線。

- コントロールパネル or API で **VM を停止**（Ollama/コンテナの stop だけでは不十分。課金を
  止めるには **VM 自体の停止**が必要）
- 停止中もディスク費は少額かかる。長期放置は避ける
- 消し忘れ防止に、使用後は毎回停止する運用を徹底

### Step 12. 撤収（使い終わったら）

```bash
docker compose down
# 必要ならデータをバックアップしてから
```

コントロールパネルから **VM とディスクを削除**。短期・使い捨て前提なので、
用が済んだら丸ごと消してディスク課金も止める。

## 5. つまずきポイント早見表

| 症状 | 原因 | 対処 |
|------|------|------|
| コンテナから Ollama に繋がらない | Linux で Ollama が 127.0.0.1 のみ LISTEN | `OLLAMA_HOST=0.0.0.0`（Step 4）|
| `host.docker.internal` 解決不可 | extra_hosts 未設定 | compose に `host-gateway`（既存で対応済み）|
| モデルが VRAM に載らない/激遅 | V100 32GB に対しモデルが大きい | H100 に切替 or 量子化を強める（Step 0）|
| SearXNG が起動拒否 | `SEARXNG_SECRET` 未設定 | `.env` を配置（Step 7）|
| web 検索が CAPTCHA で死ぬ | ddgs にフォールバック | `config.yaml` の `web.backend: searxng`（既存）|
| 想定外の高額請求 | VM 停止し忘れ | 使用後は必ず VM 停止（Step 11）|

---

## 6. （代替）Ollama もコンテナ化する場合

ホスト直起動が最小構成だが、compose に寄せたい場合は Ollama もサービス化できる。
その場合は VM に `nvidia-container-toolkit` を導入し、compose に GPU 予約を追加、
`config.yaml` の `base_url` を `http://ollama:11434/v1` に変更する。
短期・省手間なら **ホスト直起動（本手順の本線）を推奨**。
