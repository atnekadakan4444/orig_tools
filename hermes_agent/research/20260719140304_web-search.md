# Hermes Agent Web検索がCAPTCHAで弾かれる件 調査レポート

- 調査日: 2026-07-19
- 対象: `hermes_agent/docker-compose.yml`（コンテナ名 `hermes-sandbox`, image `nousresearch/hermes-agent:latest`）
- 症状: web検索させると「GoogleとDuckDuckGoの両方がCAPTCHAでブロック」という応答が返り、検索できない

## 結論（サマリ）

原因は **web検索バックエンドが未設定** なこと。Hermes のweb検索は複数バックエンドを
選べる設計だが、`config.yaml` に `web:` 節がなく、検索系APIキーのenvも無いため、
自動検出カスケードの **最下段 `ddgs`（DuckDuckGo）/ `brave-free` のHTMLスクレイピング**
にフォールバックしている。無料・キー無しのスクレイピング系は検索エンジン側の
CAPTCHA/ボット検知で弾かれるため、リトライしても復旧しない。

対策は **まともな検索バックエンドを明示指定すること**。現実的な選択肢は2つ:

- **A. SearXNG を自前ホスト**（APIキー不要・完全ローカル・無料）← 本構成に最適
- **B. 検索APIキー（Tavily / Exa / Brave）を設定**（設定は簡単だが外部SaaS依存）

## 切り分けの経緯

### 1. web検索ツールの実装場所

コンテナ内 `/opt/hermes/tools/web_tools.py` が本体。冒頭コメントより、web検索は
「複数バックエンドプロバイダで動作する汎用ツール」で、バックエンドは
`hermes tools` セットアップ時に `web.backend`（config.yaml）で選択される設計。

対応バックエンド（`_LEGACY_WEB_BACKENDS`）:

```python
{"parallel", "firecrawl", "tavily", "exa", "searxng", "brave-free", "ddgs", "xai"}
```

### 2. バックエンド自動選択カスケード

`_get_backend()` は `web.backend` が未設定の場合、env の有無で以下の優先順に自動選択:

```python
backend_candidates = (
    ("tavily",     _has_env("TAVILY_API_KEY")),
    ("exa",        _has_env("EXA_API_KEY")),
    ("parallel",   _has_env("PARALLEL_API_KEY")),
    ("firecrawl",  _has_env("FIRECRAWL_API_KEY") or _has_env("FIRECRAWL_API_URL")),
    ("firecrawl",  _is_tool_gateway_ready()),
    ("searxng",    _has_env("SEARXNG_URL")),
    ("brave-free", _has_env("BRAVE_SEARCH_API_KEY")),
    ("ddgs",       _ddgs_package_importable()),
)
# 上から順に available（=キー有り）のものを採用。全滅時の最終既定は "firecrawl"。
```

**有料/キー有りバックエンドが優先され、無料枠（brave-free, ddgs）が末尾。**
キーが1つも無いと、パッケージが入っているだけで通る `ddgs`（DuckDuckGo）に落ちる。

### 3. 現在の設定状態（コンテナ実機確認）

```
# 実行中 config.yaml の web 節
→ (web節なし = 自動検出モード)

# 検索系 env（TAVILY / EXA / PARALLEL / FIRECRAWL / SEARXNG / BRAVE / SERPER）
→ (すべて未設定)
```

→ 自動検出が末尾の `ddgs` まで落ち、DuckDuckGo のHTMLスクレイピングを実行。
これが CAPTCHA で弾かれ、ユーザーが見た「Google/DuckDuckGo両方ブロック」応答になる。

## 根本原因

web検索バックエンドが未設定のため、キー不要の無料スクレイピング系（ddgs/brave-free）に
フォールバックしている。これらは検索エンジンのボット検知（CAPTCHA）で恒常的にブロック
されるため、web検索が実質機能しない。ネットワークやモデルの問題ではない。

## 対策の選択肢

| 方式 | 内容 | 長所 | 短所 |
|---|---|---|---|
| **A. SearXNG 自前ホスト** | compose に searxng コンテナを追加し `SEARXNG_URL` を指す | APIキー不要・完全ローカル・無料・レート制限を自分で管理 | コンテナ1個追加、初期設定あり |
| **B. 検索APIキー** | Tavily / Exa / Brave のキーを取得し env で渡す | 設定が最も簡単・最も安定 | 外部SaaS依存・キー管理が必要（各社に無料枠あり） |

### 推奨: A（SearXNG 自前ホスト）

本構成は Ollama をローカル運用し、secrets をコードに置かない方針。SearXNG は
複数検索エンジンを束ねるメタ検索を自前でホストでき、APIキー不要・完全ローカルで
完結するため方針に最も合致する。

想定手順（未実施・メモ）:

1. `docker-compose.yml` に `searxng` サービスを追加（JSON出力フォーマットを有効化）
2. hermes-agent の env に `SEARXNG_URL=http://searxng:8080` を追加
   （同一 compose ネットワーク内のサービス名で到達）
3. `config.yaml` に明示指定:
   ```yaml
   web:
     backend: searxng
   ```
4. コンテナ再起動 → web検索がSearXNG経由になることを確認

### 代替: B（検索APIキー）

- `.env` に `TAVILY_API_KEY`（または EXA / BRAVE）を置き、compose の env で渡す。
- `config.yaml` に `web: {backend: tavily}` を明示（自動検出でも tavily が最優先で採用される）。
- 設定は最も簡単だが、キー管理と外部依存が発生する。

## 次のアクション（保留中）

- SearXNG採用時は `search_backend` / `extract_backend` の分離指定も可能
  （検索=SearXNG、本文抽出=Firecrawl 等）。必要に応じて検討。

## 参考

- web検索の権威は `config.yaml` の `web.backend`（`hermes tools` で設定される項目）。
- capability別に `web.search_backend` / `web.extract_backend` で検索と抽出を別プロバイダに
  分けることも可能（`_get_capability_backend()`）。
- モデル設定同様、Hermesの設定は基本 `config.yaml` が権威（前回調査 `20260719132113_research.md` 参照）。
