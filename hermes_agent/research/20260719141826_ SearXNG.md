# Hermes Agent Web検索 SearXNG化 対応レポート

- 対応日: 2026-07-19
- 対象: `hermes_agent/`（`docker-compose.yml`, `config.yaml`, `searxng/settings.yml`, `.env`）
- 目的: web検索がCAPTCHAで恒常ブロックされる問題を解消する
- 前提調査: `research/20260719140304_web-search.md`（原因の切り分け）

## 結論（サマリ）

web検索バックエンドを **自前ホストのSearXNG** に固定した。
これまではバックエンド未設定のため自動検出で無料の `ddgs`(DuckDuckGo)スクレイピングに
落ち、CAPTCHAで弾かれて検索不能だった。SearXNG（メタ検索エンジン）をcomposeに追加し、
`config.yaml` の `web.backend: searxng` と env `SEARXNG_URL` で明示指定することで解決。

- APIキー不要・完全ローカルで完結（外部SaaS非依存）
- 元々失敗していたクエリ「奴隷の鎖自慢」が実結果を返すところまでエンドツーエンド検証済み

## 修正内容

### 1. `docker-compose.yml`：searxngサービス追加とHermesへの配線

- 新規サービス `searxng`（`searxng/searxng:latest`, コンテナ名 `hermes-searxng`）を追加。
- 設定を宣言的に管理するため `./searxng:/etc/searxng` をbind mount。
- secret_key は env `SEARXNG_SECRET` で注入（`.env` 経由）。ファイルには秘密を書かない。
- デバッグ用にホストの `127.0.0.1:8888` のみ公開（Hermesからは同一composeネットワークの
  サービス名 `searxng:8080` で到達するため、ホスト公開は必須ではない）。
- hermes-agent に env `SEARXNG_URL=http://searxng:8080` を追加。
- hermes-agent に `depends_on: [searxng]` を追加し起動順を明示。

```yaml
  # hermes-agent 側
  environment:
    - SEARXNG_URL=http://searxng:8080
  depends_on:
    - searxng

  # 追加サービス
  searxng:
    image: searxng/searxng:latest
    container_name: hermes-searxng
    restart: unless-stopped
    environment:
      - SEARXNG_SECRET=${SEARXNG_SECRET}
    volumes:
      - ./searxng:/etc/searxng:rw
    ports:
      - "127.0.0.1:8888:8080"
```

### 2. `searxng/settings.yml`（新規）：JSON API有効化

SearXNG同梱の既定 `searx/settings.yml` は `search.formats: [html]` に制限されており、
HermesがJSON API（`/search?format=json`）を叩けない。`use_default_settings: true` で
既定にマージしつつ、`json` を有効化する。秘密（secret_key）はこのファイルに置かない。

```yaml
use_default_settings: true
search:
  formats:
    - html
    - json
server:
  secret_key: "ultrasecretkey"   # 実値は ${SEARXNG_SECRET} で上書き
  limiter: false                 # 単一内部利用者のためレート制限は無効
  public_instance: false
  image_proxy: false
```

### 3. `config.yaml`：バックエンドをsearxngに固定

`browser:` の直後に `web:` 節を追加。未指定だとキーの有無で自動検出され、キー無しの
本環境では `ddgs` に落ちるため、明示指定で `searxng` に固定する。

```yaml
web:
  backend: searxng
```

### 4. `.env`：secret_keyの設定（秘密はここだけに置く）

`.env`（gitignore済）に以下を追加する。値は今回 `openssl rand -hex 32` で生成した。

```
SEARXNG_SECRET=<64桁のhex文字列>
```

## Hermes側の仕様（確認済み）

- 検索バックエンドの権威は `config.yaml` の `web.backend`。capability別に
  `web.search_backend` / `web.extract_backend` で検索と抽出を別プロバイダに分けることも可能。
- SearXNGプロバイダ（`/opt/hermes/plugins/web/searxng/provider.py`）は
  env `SEARXNG_URL` を基点に `{SEARXNG_URL}/search?format=json&q=...` を httpx で叩く。
  `is_available()` は `SEARXNG_URL` がセットされていれば True。
- バックエンド自動選択の優先順（`web_tools.py` `_get_backend`）:
  `tavily → exa → parallel → firecrawl → searxng → brave-free → ddgs`。
  キーが1つも無いと最下段 `ddgs` に落ちる（＝今回の不具合の発生経路）。

## SearXNGイメージの仕様（対応の根拠）

- entrypoint は `/etc/searxng/settings.yml` が **存在しない場合のみ** テンプレートから生成し、
  その際に `ultrasecretkey` をランダム値へ置換する。settings.yml を自前で置くと置換は走らない。
  → そのため secret_key は env `SEARXNG_SECRET` で注入する方式にした。
- secret_key が既定 `ultrasecretkey` のまま非debugで起動すると `webapp.py` が
  `sys.exit(1)` する（＝secret未設定だと起動しない仕様）。
- granian が `::`（全インターフェース）:8080 でLISTENするため、
  compose内の他コンテナから `searxng:8080` で到達可能。
- 同梱既定は `formats: [html]` のみ・`limiter: false`・`public_instance: false`。

## 検証結果（実機・エンドツーエンド）

| 確認項目 | 結果 |
|---|---|
| SearXNG本体 JSON検索API（ホスト `127.0.0.1:8888`） | `HTTP 200`、31件取得 |
| Hermesコンテナ → `searxng:8080` 疎通 | `HTTP 200`、28件取得 |
| Hermesが解決するバックエンド `_get_search_backend()` | `searxng` |
| `web_search_tool("奴隷の鎖自慢")` の実呼び出し | 実結果を返す（ameblo / X / はてなブログ 等） |

起動ログに `wikidata: engine init ... HTTP 403` が出るが、これは束ねる多数エンジンのうち
1つの初期化失敗で、他エンジンで集約されるため検索結果には影響しない。

## 運用手順

### 初回セットアップ

1. `hermes_agent/.env` に `SEARXNG_SECRET=<openssl rand -hex 32 の値>` を追加。
2. スタック起動:
   ```
   cd hermes_agent
   docker compose up -d
   ```
3. 動作確認（任意）:
   - ブラウザ / curl で `http://127.0.0.1:8888/search?format=json&q=test`
   - Hermesに実際にweb検索させて結果が返ることを確認。

### 変更・メンテ

- 検索バックエンドの切替は `config.yaml` の `web.backend` を編集して再起動するだけ。
- SearXNGの設定変更は `hermes_agent/searxng/settings.yml` を編集して
  `docker compose restart searxng`。
- `SEARXNG_SECRET` をローテートする場合は `.env` を更新して `docker compose up -d`。

## 注意点

- `docker-compose.yml` の `SEARXNG_SECRET=${SEARXNG_SECRET}` は、`.env` 未設定だと
  **空文字が渡り、SearXNGは弱いsecret_keyのまま起動する**（`ultrasecretkey` ではないため
  起動拒否の `sys.exit` は回避されるが、署名鍵として不適切）。運用では必ず `.env` に
  実値を設定すること。未設定を起動時に弾きたい場合は `${SEARXNG_SECRET:?メッセージ}` に
  戻すと、compose実行時点でエラーになり気付ける。
- `secret_key` はSearXNG内部の署名/CSRF用のインスタンス鍵であり、外部SaaSの認証情報ではない。
  ただし秘密情報として扱い、コード（追跡対象ファイル）にはハードコードしない。
- `network_mode: "none"` はOllama連携（`host.docker.internal`）に加え、
  SearXNGが外部検索エンジンへ出るためにも使えない。
